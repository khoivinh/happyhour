## Flag fixes (this session, post-agent)

Two issues surfaced during browser smoke-testing:

### Flag A — appearance cycler stopped cycling icons

**Cause:** Agent misread the 2252 spec ("Use this icon for the
dark/light/system/Happy mode control") as asking for a single static icon and
collapsed `getThemeIcon()` into a static `<AppearanceIcon>` wrapper. Figma
node `198:1789` (full Happy-mode sidebar) actually shows the Appearance row's
trailing icon as the Happyhour smiley — i.e. the trailing icon **is**
per-state (monitor/sun/moon/smiley).

**Fix** in `client/src/components/sidebar.tsx`:
- Restored the `Monitor, Smartphone, Sun, Moon` lucide imports + `HappyhourLogo`
  import; dropped `AppearanceIcon`.
- Restored `getThemeIcon()` with the original `system → Monitor/Smartphone`,
  `light → Sun`, `dark → Moon`, `happy → HappyhourLogo` mapping.
- Swapped the render site back from `<AppearanceIcon />` to `{getThemeIcon()}`.
- Deleted `client/src/components/icons/appearance-icon.tsx`.

### Flag B — dark-mode logo face rendered white

**Cause:** Original component used `fill="currentColor"` on the face (smile +
checkmark-eye) paths, which picked up `text-foreground` — near-black in light
mode, near-white in dark mode. Canonical `client/public/favicon.svg` uses
`fill="black"` explicitly; Figma node `114:1332` (dark-mode header) shows the
mark with **yellow circle + black face**.

**Fix** in `client/src/components/icons/happyhour-logo.tsx`:
- Face path `fill` changed from `currentColor` to `#000000` for both paths.
- This made the `"dark"` variant identical to `"default"` (both yellow circle
  + black face), so simplified the union from `"default" | "dark" | "happy"`
  to `"default" | "happy"`.
- `client/src/pages/world-clock.tsx` — `logoVariant` logic collapsed from a
  3-way ternary to `resolvedTheme === "happy" ? "happy" : "default"`.
